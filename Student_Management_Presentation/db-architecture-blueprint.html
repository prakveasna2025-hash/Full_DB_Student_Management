Sheet 01 / 04Split-Table Pattern
01

Solving the Single-Identity Hierarchy

one master record, three specialized shells
The problem

Students, teachers, and parents are distinct roles — but they share identical human attributes: email, phone, login credentials, active status. Model each as its own table and password_hash ends up duplicated four times over, drifting out of sync the moment one copy changes.

The structural core

A one-to-one sub-typing hierarchy. Every person gets exactly one master identity row in users. Specialized tables — students, teachers, parents — hold only what's role-specific, each mapped back to that row by user_id.

Sheet 02 / 04Circular Constraint Fix
02

Eliminating Relational Deadlocks

breaking the chicken-and-egg foreign key cycle
The problem

Auto-generated constraints wired the relationship both ways at once: a user couldn't be inserted without a matching student row, and a student couldn't be inserted without a matching user row. Neither side could go first — a genuine deadlock at the schema level, surfaced as a stray constraint named users_user_id_fkey1.

The structural core

Diagnose and drop the circular foreign key, then restructure the flow linearly: users is always created first, and every profile type points strictly downward to it. No deferred transactions, no insert ordering gymnastics — the application layer just writes rows in one direction.

Fig 2.1 — the mutual FK is dropped; dependency now resolves strictly downward.
Sheet 03 / 04UUID v4 Indexing Strategy
03

Mitigating Storage Fragmentation

trading sequential disk order for indexed lookup speed
The problem

gen_random_uuid() gives every row a secure, distribution-safe identifier — but a random UUID has no relationship to insertion order. Rows land in scattered, unpredictable positions across disk pages instead of appending neatly, which is exactly what sequential IDs are good at.

The structural core

Targeted B-Tree indexes on every high-frequency foreign key — attendance, fees, submissions. Rows can still live anywhere on disk; the index guarantees the search path stays O(log n) instead of degrading into a full table scan.

Sheet 04 / 04OLTP → OLAP
04

Transitioning From Operations to Analytics

flattening a normalized schema into one queryable view
The problem

A normalized transactional schema is deliberately split across dozens of tables to avoid duplication — good for writes, hostile to reads. A dashboard asking "how is this student doing?" would otherwise need a seven-table JOIN just to answer a basic question.

The structural core

A Master Analytical View — analytics_student_master_dashboard — flattens the operational layout into one virtual matrix. It computes attendance percentage, normalized assignment averages, and net outstanding balance on the fly, ready for a BI tool or a Python pipeline to consume directly.
